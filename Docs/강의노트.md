# 강의노트

## Windows API 블랙박스

Windows API 함수 내부는 `F11`로 들어가지 못하는 경우가 많다. 예를 들어 `ShowWindow`, `CreateWindowW` 같은 함수는 Microsoft가 제공하는 API 함수이기 때문에 내부 구현을 직접 볼 수 없거나, 볼 수 있더라도 우리가 수정할 수 없다.

Windows API는 내부 구현을 모두 볼 수 없고, 볼 수 있더라도 전부 분석하기 어렵다.

기존 C++ 학습에서는 개념을 먼저 배우고, 코드를 직접 보면서 이해한 뒤 문제를 풀었다. 하지만 API나 게임 엔진, 라이브러리를 사용할 때는 접근 방식이 조금 달라져야 한다.

API 학습에서는 다음이 더 중요하다.

- 어떤 함수가 있는지 안다.
- 어떤 인자를 넣어야 하는지 안다.
- 반환값이 무엇인지 안다.
- 입력을 바꾸면 결과가 어떻게 달라지는지 실험한다.
- 내부 구현을 몰라도 사용할 수 있게 익숙해진다.

게임 개발에서는 Unity, Unreal, DirectX, Windows API처럼 내부를 전부 알 수 없는 도구를 많이 사용한다. 따라서 모든 내부를 완벽히 이해한 뒤에 사용하는 방식으로는 진행이 느려진다.

처음에는 어색하더라도, "어떤 값을 넣으면 어떤 결과가 나온다"는 식으로 사용하면서 익숙해져야 한다.

## `framework.h`와 헤더 파일

기본 프로젝트의 상단에는 `framework.h` 같은 파일이 포함되어 있다.

이 파일은 여러 라이브러리 관련 헤더들을 모아 놓은 파일이다. 콘솔 프로젝트에서 `#include <iostream>`을 사용했던 것처럼, 필요한 헤더들을 포함하기 위해 사용한다.

## 핸들이란?

핸들은 실제 메모리 주소처럼 보이는 값일 수도 있고, 운영체제가 내부적으로 관리하는 테이블의 식별자일 수도 있다.

중요한 것은 사용자가 내부 메모리를 직접 조작하지 않고, Windows API 함수에 핸들을 넘겨서 작업한다는 점이다.

예를 들어 창을 만들고 나면 Windows는 그 창에 접근할 수 있는 핸들을 돌려준다.

```cpp
HWND hWnd = CreateWindowW(...);
```

이후 창을 보여주거나 닫거나 업데이트할 때는 이 `hWnd`를 함수에 넘긴다.

```cpp
ShowWindow(hWnd, nCmdShow);
UpdateWindow(hWnd);
DestroyWindow(hWnd);
```

즉, 핸들은 Windows API를 사용할 때 "내가 어떤 대상에 작업을 할 것인지" 알려주는 값이다.

## Windows 창 생성 흐름 복습

이번 강의 초반에는 이전 시간에 배운 Windows 창 생성 흐름을 다시 복습했다.

Windows API 코드는 함수도 많고 매개변수도 많다. 처음부터 모든 매개변수의 정확한 의미를 외우려고 하면 오히려 진행이 막힌다. API를 공부할 때는 당장 중요하지 않은 부분은 과감하게 넘기고, 지금 이해해야 하는 핵심 흐름부터 잡는 것이 중요하다.

Windows 창 하나가 실행되는 큰 흐름은 다음 네 가지로 정리할 수 있다.

1. 윈도우 클래스 정보를 등록한다.
2. 등록된 정보를 이용해 실제 윈도우를 만든다.
3. 만든 윈도우를 화면에 보여준다.
4. 메시지 루프를 돌면서 프로그램을 계속 실행한다.

이 네 가지가 가장 중요하고, 나머지 세부 매개변수는 필요할 때 찾아보면 된다.

## `MyRegisterClass`와 윈도우 정보 등록

`MyRegisterClass`는 실제 창을 만들기 전에 "이런 모양의 창을 만들겠다"는 정보를 Windows에 등록하는 단계다.

이 함수 안에서는 `WNDCLASSEXW` 구조체에 창 정보를 채운다. 여기에는 다음과 같은 정보가 들어간다.

- 윈도우 클래스 이름
- 아이콘
- 커서 모양
- 배경색
- 메뉴 정보
- 메시지 처리 함수

이 단계는 실제 창을 만드는 단계가 아니다. 메모리 어딘가에 "Editor Window라는 이름의 창은 이런 정보를 가진다"라고 등록해 두는 단계다.

나중에 `CreateWindowW`가 호출될 때 이 이름을 기준으로 등록된 정보를 찾아 실제 창을 만든다.

## `CreateWindowW`와 창 핸들

`CreateWindowW`는 등록된 윈도우 클래스 정보를 바탕으로 실제 윈도우 창을 만든다.

이때 앞에서 등록한 윈도우 클래스 이름이 인자로 들어간다. Windows는 이 이름을 보고 미리 등록된 창 정보를 찾아온다. 그리고 그 정보를 기반으로 실제 창에 필요한 메모리를 할당한다.

창을 만들 때는 위치, 크기, 스타일 같은 인자도 함께 전달된다. 하지만 지금 단계에서 모든 인자의 의미를 외울 필요는 없다. 중요한 것은 `CreateWindowW`가 실제 창을 만들고, 성공하면 그 창에 접근할 수 있는 핸들을 반환한다는 점이다.

```cpp
HWND hWnd = CreateWindowW(...);
```

`hWnd`는 생성된 창을 가리키는 핸들이다. 이후 `ShowWindow`, `UpdateWindow`, `DestroyWindow` 같은 함수에 이 핸들을 넘겨서 특정 창을 조작한다.

## `ShowWindow`, `UpdateWindow`, 메시지 루프

`CreateWindowW`로 창을 만들었다고 해서 바로 화면에 보이는 것은 아니다. 생성된 창을 실제 화면에 보여주려면 `ShowWindow`를 호출해야 한다.

```cpp
ShowWindow(hWnd, nCmdShow);
```

그 다음 `UpdateWindow`로 창을 한 번 갱신한다.

```cpp
UpdateWindow(hWnd);
```

이후 프로그램은 메시지 루프에 들어간다. 프로그램이 바로 종료되지 않고 계속 켜져 있으려면 반복문이 계속 돌아야 한다.

Windows 프로그램에서는 이 반복문 안에서 메시지 큐의 메시지를 하나씩 꺼내 처리한다.

메시지는 다음과 같은 요청이나 이벤트를 의미한다.

- 창을 닫아라
- 마우스를 클릭했다
- 키보드를 눌렀다
- 창을 다시 그려라
- 창이 이동했다

이 메시지들을 처리하는 함수가 `WndProc`이다. `WndProc`은 윈도우 클래스 정보를 등록할 때 함께 지정된다.

즉, 창 생성 흐름은 다음처럼 이어진다.

```cpp
MyRegisterClass(...); // 창 정보 등록
CreateWindowW(...);   // 등록된 정보로 창 생성
ShowWindow(...);      // 화면에 표시
UpdateWindow(...);    // 창 갱신
while (...)           // 메시지 루프
{
    ...
}
WndProc(...);         // 메시지 처리
```

## API 학습법

Windows API를 공부할 때는 코드 전체를 한 번에 완벽히 이해하려고 하면 안 된다.

API에는 매개변수가 많고 내부 구현을 볼 수 없는 함수도 많다. 모든 것을 이해하고 넘어가려고 하면 API만 오래 공부하게 되고, 정작 목표인 게임 제작으로 넘어가지 못한다.

강의에서는 수학 공부에 비유했다. 집합을 완벽히 이해하지 못했다고 계속 집합만 반복하면 뒤의 함수, 도형, 미분, 적분으로 넘어가지 못한다. 공부는 한 단원을 완벽히 끝낸 뒤 다음으로 가는 방식보다, 전체 흐름을 여러 번 반복하면서 점점 이해도를 올리는 방식이 더 효율적일 때가 많다.

API 공부도 마찬가지다.

- 지금 필요한 큰 흐름을 먼저 이해한다.
- 당장 중요하지 않은 매개변수는 넘긴다.
- 필요한 순간에 문서나 노트를 찾아본다.
- 값을 바꿔 보며 결과를 확인한다.
- 전체를 여러 번 반복하면서 익숙해진다.

현재 목표는 Windows API 자체를 연구하는 것이 아니라, API를 이용해 게임 엔진을 만드는 것이다.

## `WM_PAINT`와 그리기 작업

이번 강의 후반에는 `WM_PAINT` 메시지 안에서 화면에 도형을 그리는 방법을 다뤘다.

`WM_PAINT`는 창을 다시 그려야 할 때 들어오는 메시지다. 예를 들어 창이 처음 나타나거나, 가려졌다가 다시 보이거나, 갱신이 필요할 때 호출될 수 있다.

기본 코드에서는 다음 흐름으로 그리기 작업을 시작한다.

```cpp
PAINTSTRUCT ps;
HDC hdc = BeginPaint(hWnd, &ps);
// 그리기 작업
EndPaint(hWnd, &ps);
```

여기서 중요한 개념이 `DC`다.

## DC란?

`DC`는 Device Context의 약자다. 화면에 그리기 위해 필요한 정보를 담고 있는 데이터 구조라고 이해하면 된다.

윈도우의 크기나 메뉴 같은 창 자체를 조작할 때는 윈도우 핸들인 `HWND`를 사용한다. 반면 화면에 선, 사각형, 원, 글자 등을 그리는 작업은 `DC`를 통해 진행한다.

DC에는 화면 출력에 필요한 정보들이 들어 있다.

- 어떤 색으로 그릴지
- 어떤 선 굵기를 사용할지
- 어떤 브러시를 사용할지
- 어떤 펜을 사용할지
- 어떤 폰트를 사용할지

글자도 화면에 출력되는 순간에는 결국 하나의 그림처럼 처리된다. 그래서 글씨를 그릴 때도 DC와 관련된 정보가 사용된다.

정리하면 다음과 같다.

- `HWND`: 어떤 윈도우를 조작할지 가리키는 핸들
- `HDC`: 어떤 윈도우의 화면에 어떻게 그릴지 다루는 핸들

## `BeginPaint`와 `EndPaint`

`BeginPaint`는 그리기 작업을 시작하면서 DC를 얻는 함수다.

```cpp
HDC hdc = BeginPaint(hWnd, &ps);
```

인자로는 윈도우 핸들인 `hWnd`가 들어간다. Windows는 이 핸들을 보고 어떤 창에 그릴 것인지 판단하고, 그 창에 대한 DC를 반환한다.

그리기 작업이 끝나면 반드시 `EndPaint`를 호출해야 한다.

```cpp
EndPaint(hWnd, &ps);
```

즉, `WM_PAINT` 안에서는 보통 다음 순서를 지킨다.

1. `BeginPaint`로 DC를 얻는다.
2. DC를 이용해 도형이나 글자를 그린다.
3. `EndPaint`로 그리기 작업을 끝낸다.

## 사각형 그리기: `Rectangle`

Windows API에는 사각형을 그리는 `Rectangle` 함수가 있다.

```cpp
Rectangle(hdc, left, top, right, bottom);
```

첫 번째 인자는 `HDC`다. 어떤 창의 화면에 그릴지 알려주는 값이다.

나머지 네 값은 사각형의 위치와 크기를 정한다.

- `left`: 왼쪽 X 좌표
- `top`: 위쪽 Y 좌표
- `right`: 오른쪽 X 좌표
- `bottom`: 아래쪽 Y 좌표

Windows 화면 좌표는 왼쪽 위가 `(0, 0)`이다.

예를 들어 다음처럼 호출하면 왼쪽 위 기준으로 `(100, 100)`부터 `(200, 200)`까지의 사각형을 그린다.

```cpp
Rectangle(hdc, 100, 100, 200, 200);
```

이때 `right`, `bottom`은 너비와 높이가 아니라 오른쪽 아래 좌표로 이해하는 것이 좋다.

## 원 그리기: `Ellipse`

원 또는 타원을 그릴 때는 `Ellipse` 함수를 사용할 수 있다.

```cpp
Ellipse(hdc, left, top, right, bottom);
```

`Ellipse`도 `Rectangle`과 비슷하게 왼쪽 위 좌표와 오른쪽 아래 좌표를 사용한다.

일반적으로 원을 생각할 때는 중심 좌표와 반지름을 떠올리지만, Windows API의 `Ellipse`는 그런 방식이 아니다. 먼저 사각형 영역을 잡고, 그 사각형 안에 들어가는 타원을 그린다고 이해하면 된다.

예를 들어 다음 코드는 `(200, 200)`부터 `(300, 300)`까지의 사각형 영역 안에 원을 그린다.

```cpp
Ellipse(hdc, 200, 200, 300, 300);
```

API는 우리가 일반적으로 떠올리는 방식과 다르게 설계되어 있을 수 있다. 중요한 것은 API가 요구하는 방식에 맞춰 사용하는 것이다.

## GDI 오브젝트

GDI는 Windows에서 2D 그래픽 출력을 다루는 오래된 그래픽 시스템이다.

DC 안에는 기본적으로 그리기에 필요한 여러 GDI 오브젝트가 선택되어 있다. 예를 들어 기본 상태에서는 흰색 배경 브러시와 검은색 선 펜이 선택되어 있을 수 있다.

그래서 별도로 색을 지정하지 않아도 사각형을 그리면 흰색 내부와 검은색 테두리로 그려진다.

대표적인 GDI 오브젝트는 다음과 같다.

- 브러시: 면을 칠할 때 사용
- 펜: 선을 그릴 때 사용
- 폰트: 글자를 그릴 때 사용

## 브러시로 면 색 바꾸기

브러시는 도형 내부를 칠하는 데 사용한다.

예를 들어 파란색 브러시를 만들려면 `CreateSolidBrush`를 사용할 수 있다.

```cpp
HBRUSH blueBrush = CreateSolidBrush(RGB(0, 0, 255));
```

하지만 브러시를 만들었다고 해서 바로 그 브러시가 사용되는 것은 아니다. DC에 선택해야 한다.

```cpp
HBRUSH oldBrush = (HBRUSH)SelectObject(hdc, blueBrush);
```

`SelectObject`는 새 오브젝트를 DC에 선택하면서, 이전에 선택되어 있던 오브젝트를 반환한다.

이전 오브젝트를 받아 두는 이유는 나중에 원래 상태로 되돌리기 위해서다.

기본 흐름은 다음과 같다.

```cpp
HBRUSH blueBrush = CreateSolidBrush(RGB(0, 0, 255));
HBRUSH oldBrush = (HBRUSH)SelectObject(hdc, blueBrush);

Rectangle(hdc, 100, 100, 200, 200);

SelectObject(hdc, oldBrush);
DeleteObject(blueBrush);
```

순서가 중요하다.

1. 새 브러시를 만든다.
2. 새 브러시를 DC에 선택한다.
3. 도형을 그린다.
4. DC를 이전 브러시로 되돌린다.
5. 새로 만든 브러시를 삭제한다.

## 왜 원래 브러시로 되돌려야 하는가?

`SelectObject`로 DC의 브러시를 바꾸면 이후 그리기 작업에도 계속 그 브러시가 적용될 수 있다.

그래서 특정 도형만 파란색으로 그리고 싶다면, 그리기가 끝난 뒤 원래 브러시로 되돌려야 한다.

원래 브러시를 저장하지 않으면 되돌릴 방법이 사라질 수 있다.

```cpp
HBRUSH oldBrush = (HBRUSH)SelectObject(hdc, blueBrush);
```

이 코드는 "새 파란 브러시를 선택하고, 이전에 선택되어 있던 브러시를 oldBrush에 저장한다"는 의미다.

그 다음 그리기가 끝나면 다음처럼 되돌린다.

```cpp
SelectObject(hdc, oldBrush);
```

## 만든 GDI 오브젝트는 삭제해야 한다

`CreateSolidBrush`나 `CreatePen`처럼 직접 만든 GDI 오브젝트는 사용 후 삭제해야 한다.

```cpp
DeleteObject(blueBrush);
```

삭제하지 않으면 `WM_PAINT`가 호출될 때마다 새 브러시가 계속 만들어지고, 메모리에 남을 수 있다. 이것은 메모리 누수로 이어질 수 있다.

주의할 점은 `delete`로 지우는 것이 아니라, Windows API가 제공하는 `DeleteObject`를 사용해야 한다는 것이다.

GDI 오브젝트를 사용할 때의 안전한 순서는 다음과 같다.

1. 만든다.
2. DC에 선택한다.
3. 그린다.
4. 원래 오브젝트로 되돌린다.
5. 만든 오브젝트를 삭제한다.

## 펜으로 선 색과 굵기 바꾸기

펜은 도형의 테두리나 선을 그릴 때 사용한다.

펜은 `CreatePen`으로 만들 수 있다.

```cpp
HPEN redPen = CreatePen(PS_SOLID, 2, RGB(255, 0, 0));
```

인자는 보통 다음 의미를 가진다.

- 첫 번째 인자: 선 스타일
- 두 번째 인자: 선 두께
- 세 번째 인자: 선 색상

예를 들어 `PS_SOLID`는 일반 실선이다. 그 외에도 점선이나 대시 선 같은 스타일이 있다.

펜도 브러시와 사용 방식이 같다.

```cpp
HPEN redPen = CreatePen(PS_SOLID, 2, RGB(255, 0, 0));
HPEN oldPen = (HPEN)SelectObject(hdc, redPen);

Ellipse(hdc, 200, 200, 300, 300);

SelectObject(hdc, oldPen);
DeleteObject(redPen);
```

이렇게 하면 빨간색, 두께 2의 선으로 도형의 테두리를 그릴 수 있다.

펜도 직접 만들었다면 반드시 `DeleteObject`로 삭제해야 한다.

## 스톡 오브젝트

자주 사용하는 GDI 오브젝트는 Windows가 미리 만들어 둔다. 이런 기본 제공 오브젝트를 스톡 오브젝트라고 한다.

예를 들어 다음과 같은 것들이 있다.

- 검은색 브러시
- 흰색 브러시
- 회색 브러시
- 투명 브러시
- 검은색 펜
- 흰색 펜
- 투명 펜

스톡 오브젝트는 직접 `CreateSolidBrush`나 `CreatePen`으로 만들 필요 없이 `GetStockObject`로 가져올 수 있다.

```cpp
HBRUSH grayBrush = (HBRUSH)GetStockObject(GRAY_BRUSH);
```

그리고 일반 GDI 오브젝트처럼 DC에 선택해서 사용한다.

```cpp
HBRUSH oldBrush = (HBRUSH)SelectObject(hdc, grayBrush);
Rectangle(hdc, 400, 400, 500, 500);
SelectObject(hdc, oldBrush);
```

스톡 오브젝트는 Windows가 관리하는 기본 오브젝트이므로 직접 `DeleteObject`로 삭제하면 안 된다.

하지만 사용 후 DC를 원래 오브젝트로 되돌리는 습관은 그대로 유지하는 것이 안전하다.

## GDI 오브젝트 사용 규칙 정리

직접 만든 GDI 오브젝트와 스톡 오브젝트는 처리 방식이 조금 다르다.

직접 만든 오브젝트:

```cpp
HBRUSH brush = CreateSolidBrush(RGB(0, 0, 255));
HBRUSH oldBrush = (HBRUSH)SelectObject(hdc, brush);

Rectangle(hdc, 100, 100, 200, 200);

SelectObject(hdc, oldBrush);
DeleteObject(brush);
```

스톡 오브젝트:

```cpp
HBRUSH brush = (HBRUSH)GetStockObject(GRAY_BRUSH);
HBRUSH oldBrush = (HBRUSH)SelectObject(hdc, brush);

Rectangle(hdc, 400, 400, 500, 500);

SelectObject(hdc, oldBrush);
```

핵심 차이는 다음이다.

- 직접 만든 오브젝트는 사용 후 `DeleteObject`로 삭제한다.
- 스톡 오브젝트는 직접 삭제하지 않는다.
- 둘 다 사용 후 기존 오브젝트로 되돌리는 것이 좋다.

## 이번 강의 핵심 정리

이번 강의의 핵심은 두 가지다.

첫 번째는 Windows API 학습 태도다. 모든 매개변수와 내부 구현을 처음부터 완벽히 이해하려고 하지 말고, 큰 흐름을 먼저 잡아야 한다.

두 번째는 DC와 GDI 오브젝트다. 화면에 무언가를 그릴 때는 윈도우 핸들만 사용하는 것이 아니라, DC를 얻고 그 DC에 펜이나 브러시 같은 GDI 오브젝트를 선택해서 그린다.

기억해야 할 내용은 다음과 같다.

- Windows 창 생성의 핵심 흐름은 등록, 생성, 표시, 메시지 루프다.
- `WndProc`은 메시지를 처리하는 함수다.
- `WM_PAINT`는 창을 다시 그릴 때 들어오는 메시지다.
- `BeginPaint`는 그리기용 DC를 얻는다.
- `EndPaint`는 그리기 작업을 끝낸다.
- `Rectangle`은 사각형을 그린다.
- `Ellipse`는 사각형 영역 안에 원 또는 타원을 그린다.
- 브러시는 면 색을 담당한다.
- 펜은 선 색과 굵기를 담당한다.
- 직접 만든 GDI 오브젝트는 사용 후 `DeleteObject`로 삭제해야 한다.
- `SelectObject`를 사용했다면 원래 오브젝트로 되돌리는 습관이 필요하다.
- 스톡 오브젝트는 미리 만들어진 기본 GDI 오브젝트이며 직접 삭제하지 않는다.
